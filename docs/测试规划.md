## 一、测试概述

### 1.1 测试目标

本文档旨在对**高校学科竞赛报名管理系统**进行系统性测试规划，通过全面的测试方法、测试用例设计与缺陷追踪方案，确保系统功能完备、性能稳定、安全可靠，能够满足高校竞赛管理的实际使用需求。

具体目标如下：

- **功能正确性**：验证系统全部功能模块（用户认证、竞赛管理、报名管理、团队管理、审核流程、数据统计等）的正确运行
- **权限严密性**：验证游客、学生、教师、管理员四类角色的权限隔离与访问控制
- **异常健壮性**：验证边界条件、错误输入、异常流程的友好处理
- **数据一致性**：验证报名状态流转、软删除逻辑、事务回滚的正确性
- **安全合规性**：验证密码加密、XSS 防护、SQL 注入防护、登录限流等安全措施

### 1.2 测试范围

### 2.1 单元测试

#### 2.1.1 前端单元测试（Vitest）

| `src/api/__tests__/auth.spec.ts` | 12 | 认证 API 接口类型、请求参数验证 |

**示例测试代码：**

```typescript
// 用户状态管理测试
describe('useUserStore', () => {
  it('未登录时 isLoggedIn 应为 false', () => {
    const store = useUserStore()
    expect(store.isLoggedIn).toBe(false)
  })
  
  it('role=STUDENT 时 isStudent 为 true', () => {
    store.user = { id: 1, role: 'STUDENT' }
    expect(store.isStudent).toBe(true)
  })
})
```

```bash
cd cms-frontend

npm run test:watch # 监听模式
```

## 三、测试用例设计与方案

### 3.1 等价类划分

**（1）竞赛报名状态——等价类**

| 等级类 | 输入条件 | 预期结果 |
|--------|----------|----------|
| 有效等价类 | 当前时间在 register_start 与 register_end 之间 | 可正常报名 |
| 无效等价类 1 | 当前时间 < register_start | 显示"报名未开始" |
| 无效等价类 2 | 当前时间 > register_end | 显示"报名已截止" |

**（2）用户名注册——等价类**

| 等级类 | 输入条件 | 预期结果 |
|--------|----------|----------|
| 有效等价类 | 3~20 位字母/数字/下划线 | 注册成功 |
| 无效等价类 1 | 少于 3 位 | 提示"用户名过短" |
| 无效等价类 2 | 已存在的用户名 | 提示"用户名已存在" |
| 无效等价类 3 | 含特殊字符 | 提示"格式非法" |

**（3）密码登录——等价类**

| 等级类 | 输入条件 | 预期结果 |
|--------|----------|----------|
| 有效等价类 | 密码正确 | 登录成功 |
| 无效等价类 1 | 密码错误 | 提示"密码错误" |
| 无效等价类 2 | 密码为空 | 提示"密码不能为空" |
| 无效等价类 3 | 连续 5 次错误 | 账号锁定 15 分钟 |

### 3.2 边界值分析

**（1）团队人数限制——边界值**

| 测试点 | 输入 | 预期结果 |
|--------|------|----------|
| 下边界-1 | 1 人 | 提示"人数不足，无法提交" |
| 下边界 | 2 人 | 可正常提交报名 |
| 下边界+1 | 3 人 | 可正常提交报名 |
| 上边界 | 5 人 | 可正常提交，且达到人数上限提示 |
| 上边界+1 | 6 人 | 提示"已达到团队成员上限" |

**（2）密码长度限制——边界值**

| 测试点 | 输入 | 预期结果 |
|--------|------|----------|
| 边界-1 | 5 位 | 提示"密码至少 6 位" |
| 边界值 | 6 位 | 注册成功 |
| 边界+1 | 7 位 | 注册成功 |
| 长值 | 20 位 | 注册成功 |

**（3）登录失败计数——边界值**

| 测试点 | 输入 | 预期结果 |
|--------|------|----------|
| 边界-1 | 4 次错误 | 可继续尝试 |
| 边界值 | 5 次错误 | 第 5 次仍可尝试，但第 6 次锁定 |
| 边界+1 | 6 次尝试 | 提示账号已锁定 15 分钟 |

### 3.3 流程分析法

**流程 1——正常报名流程：**

```
学生登录 → 浏览竞赛 → 选择竞赛 → 填写信息 → 提交 → 系统校验 → 报名成功
```

**流程 2——团队报名异常流程：**

```
队长创建团队（2人） → 邀请成员 → 成员加入（3人） → 尝试提交报名
→ 系统校验：未达最小人数（min=5） → 提示"团队人数不足"
→ 继续邀请 → 满 5 人 → 提交成功
```

**流程 3——重复报名拦截：**

```
学生 A 报名竞赛 X → 报名成功 → 再次报名同一竞赛
→ 系统检测已存在记录 → 提示"已报名该竞赛" → 跳转报名记录页
```

**流程 4——登录锁定流程：**

```
用户连续错误 5 次 → 第 6 次尝试 → 系统显示"登录失败次数过多，请 15 分钟后重试"
→ 暂停尝试 → 15 分钟后 → 恢复正常登录
```
